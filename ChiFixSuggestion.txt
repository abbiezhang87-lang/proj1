#1  User schema 的 pre('save') 钩子没有调用 next()
────────────────────────────────────────────────────────────────
文件:  backend/models/User.js
原版:  第 13–25 行（共 33 行）
新版:  第 47–60 行（共 70 行）

问题：
  原代码的 pre('save') 钩子里只在 "密码没有被修改" 的分支
  early-return，正常流程（哈希完新密码后）根本没有调用 next()。
  Mongoose 的中间件约定：必须显式调 next() 或 return，否则
  save() 的 Promise 永远 pending —— 表现就是：注册接口卡死，
  一直 loading，数据根本写不进 MongoDB。

修改：
  1. 补上 next() 让控制权交给 Mongoose 继续执行
  2. 顺手整理逻辑顺序：先补默认头像 → 再判断密码是否修改
     → 需要加密才加密，否则直接 next()

影响：
  这是最要命的 bug。没修复之前连注册第一步都过不去。

────────────────────────────────────────────────────────────────
#2  登录时没有 .select('+password')，永远拿不到哈希
────────────────────────────────────────────────────────────────
文件:  backend/controllers/auth.js
原版:  第 9 行  (User.findOne({ email }))
新版:  第 87 行 (User.findOne({ email }).select('+password'))

问题：
  User schema 里 password 字段设置了 select: false（这是为了
  安全——避免任何查询不小心把密码哈希泄露出去）。
  login 函数若只用 findOne({ email })，查出来的 user.password
  是 undefined，user.matchPassword(...) 里 bcrypt.compare 拿
  undefined 比对，永远返回 false → "Invalid password"。

修改：
  加上 .select('+password') 显式"这次查询要带上密码字段做比对"。
  同时把错误提示统一成 "Invalid email or password"，防止
  账号枚举攻击。

影响：
  原代码没修复前，登录永远失败。

#5  backend/middleware/errorHandler.js
────────────────────────────────────────────────────────────────
文件:  backend/middleware/errorHandler.js
原版:  0 行
新版:  1–56 行

问题：
  没有自定义错误中间件时，所有抛错都走 Express 默认处理 ——
  返回 HTML 错误页而非 JSON，前端 axios 拿到的是一团 HTML，
  try/catch 里 err.response.data.message 永远是 undefined。
  Schema 校验错误会被当成 500 而不是 400。

修改：
  1. 统一返回 JSON 格式 { message, stack? }
  2. 特殊处理 Mongoose ValidationError → 400
  3. 特殊处理 Mongoose CastError（坏 ObjectId）→ 400
  4. 特殊处理 duplicate key（code 11000）→ 409
  5. 生产环境不返回 stack，防止信息泄露
  6. 新增 notFound 作为 404 兜底

影响：
  没修复前前端体验非常糟糕（错误提示全是 "Request failed"）。

#9  JWT 没有 expiresIn，token 永不过期
────────────────────────────────────────────────────────────────
文件:  backend/utils/generateToken.js
原版:  jwt.sign(payload, JWT_SECRET)   （没有第三个参数）
新版:  第 14–16 行 jwt.sign(payload, JWT_SECRET, { expiresIn: '7d' })

问题：
  永久有效的 JWT 一旦泄露无法失效 —— 即便用户改密码、账号被禁，
  老 token 仍能通过验证，是严重的安全隐患。

修改：
  默认 7 天过期，可通过环境变量 JWT_EXPIRES_IN 覆盖。
  配合前端 401 拦截器（见 #14），过期后自动踢回登录页。
